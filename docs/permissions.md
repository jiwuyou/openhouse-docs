# 权限与授权

本文档说明 OpenHouse 需要哪些权限、哪些权限是可选增强，以及缺失时应该如何降级。

## 权限分类

| 权限或授权 | 类型 | 用途 | 缺失时降级 |
| --- | --- | --- | --- |
| 后台运行 / 电池优化豁免 | 推荐必需 | 保持前台使用时 service-manager 和核心服务稳定。 | 提示用户可能被系统杀后台；前台仍应尽力恢复。 |
| 通知权限 | 推荐 | 显示长期运行、安装进度和失败提醒。 | UI 内显示状态，通知不可用。 |
| 存储或文件访问 | 可选增强 | 文件中转站、导入导出、让 AI 处理外部文件。 | 仅使用 App/Termux 内部目录。 |
| Shizuku | 可选高级 | Android 侧增强控制、部分 App/系统操作。 | 禁用 Shizuku 相关能力，不影响首次 Claude/Codex 可用。 |
| 无障碍 | 可选高级 | 可能用于真机 UI 自动化。 | 禁用自动操控 UI，仅保留用户手动操作。 |
| 网络访问 | 必需 | 下载、模型请求、联网检索。 | 使用内置 payload；需要下载或模型请求时提示网络问题。 |

## 首次安装要求

首次安装不应把可选高级权限作为阻塞项。必须优先保证：

- 基础环境安装。
- 文档同步。
- pi-agent 可用。
- service-manager 可用。
- Claude 或 Codex 至少一个可配置并实测。

Shizuku、无障碍、外部文件访问都不应该阻塞首次使用。

## 用户提示原则

权限提示应说明：

- 这个权限用来做什么。
- 不给权限会失去什么能力。
- 是否影响首次使用。
- 后续在哪里重新授权。

不要把所有权限堆在首次教学里。首次教学只提示必要权限和明确下一步。

## AI 操作规则

AI 发现权限缺失时，应：

1. 说明缺失权限影响哪项能力。
2. 判断是否影响当前目标。
3. 如果不影响 Claude/Codex 首次可用，先继续主线。
4. 如果影响当前目标，请用户进入对应授权页面。

AI 不应把 Shizuku 或无障碍缺失误判为 OpenHouse 安装失败。
