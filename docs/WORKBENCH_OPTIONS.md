# 主工作台选择

OpenHouse 不把任何单一界面规定为用户必须长期使用的主工作台。`pi-agent` 的主要职责是首次配置、读取文档、迁移模型配置和帮助用户理解系统。真正的主工作台由用户决定。

用户可以选择 Claude Code、Codex、Hermes Web，也可以让 AI 搜索、安装和改造其它开源项目，形成自己的长期工作空间。

## pi-agent 的定位

`pi-agent` 是：

- 首次配置助手。
- OpenHouse 文档索引员。
- pi-web 模型配置到 CloudCLI / Claude Code 的迁移执行者。
- 帮用户理解 OpenHouse 能力的引导员。
- 可选的临时任务执行入口。

`pi-agent` 不是：

- 唯一主工作台。
- 必须长期停留的聊天界面。
- 所有任务都必须经过的控制中心。
- 替代 Claude Code、Codex、Hermes Web 或其它工作台的入口。

## 可选主工作台

| 工作台 | 适合场景 | 注意事项 |
| --- | --- | --- |
| Claude Code / CloudCLI | 编程、项目修改、长任务协作、网页操作 | 需要模型配置正确，建议完成连通测试后使用 |
| Codex | 命令行编程、项目构建、测试和代码修改 | 默认在 Ubuntu 项目目录运行 |
| Hermes Web | 可选高级 Web 工作台 | 不进入 APK 核心 payload，安装耗时较久，应使用独立 uv 环境 |
| pi-agent | 首次配置、文档检索、模型迁移、系统理解 | 不应被写成唯一主工作台 |
| 其它开源项目 | 写作、知识库、项目管理、自动化、个人系统 | 需要 AI 搜索、评估、安装、注册服务和侧边栏入口 |

## “许愿，心想事成”的产品方式

用户可以直接描述想要的工作方式，例如：

- 我想要一个写作工作台。
- 我想要一个项目管理系统。
- 我想要一个个人知识库。
- 我想要一个手机上的编程平台。
- 我想要一个能管理任务、资料、日程和复盘的人生管理系统。
- 我想找一个开源 Web 项目，把它装到 OpenHouse 里长期使用。

AI 应该先问清楚需求，再检索和比较可行项目，然后说明安装成本、资源占用、维护风险和接入方式。不要一上来就安装。

## 安装开源工作台的一般流程

1. 问清用户目标、数据类型、使用频率、离线需求、隐私要求和手机性能限制。
2. 联网检索候选开源项目，优先看官方 README、文档、release、issue 和活跃度。
3. 确认运行方式：Node、Python、uv、Docker 替代方案、静态站点或单二进制。
4. 判断是否适合 Android + Termux + Ubuntu 环境。
5. 在 Ubuntu 中安装到独立目录，避免污染 OpenHouse 核心环境。
6. 写 service-manager 服务定义。
7. 写 OpenHouse 侧边栏组件注册。
8. 启动并测试本地端口。
9. 通过内置浏览器或外部浏览器确认可访问。
10. 写卸载和恢复说明。

## 接入原则

- 长期后台进程必须交给 service-manager，不要用散落的 `nohup` 常驻。
- UI 入口只注册入口、标题、图标、URL 和服务引用，不要把 shell 命令写进组件注册。
- 不确定配置格式时先检索，不硬猜。
- 涉及 API key、token、cookie 时不要写进仓库、APK 资源、公共文档、日志或截图。
- 若项目需要 GitHub 下载，网络慢时阅读 `GITHUB_NETWORK_MIRRORS.md`。
