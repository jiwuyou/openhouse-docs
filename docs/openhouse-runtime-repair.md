# 运行修复

本文定义 OpenHouse 的运行修复入口。运行修复面向普通用户、pi-agent 和维护人员，目标是在不要求用户学习终端的情况下恢复控制中枢和核心服务。

## 修复入口

必须提供用户可见的运行修复入口：

- 安装完成页。
- 主菜单或设置页。
- 服务控制页。
- 检测到 service-manager 连续失败后的错误页。
- `pi-agent` 或 `cc/codex` 不可用时的提示页。

入口名称建议：

```text
修复运行控制
```

普通用户看到的是状态和按钮；详细命令和日志只在“查看详情”中展示。

## 修复目标

运行修复按优先级恢复：

1. service-manager 控制中枢。
2. 服务注册表。
3. 默认长期服务：`smallphone`、`pi-agent`、`cloudcli`。
4. 端口和健康检查。
5. Android UI 状态同步。

运行修复不是重装系统。除非用户明确确认，不得删除用户数据、模型配置、项目、日志或已安装 payload。

## 用户可见状态

UI 应显示简短状态：

| 状态 | 含义 | 推荐动作 |
| --- | --- | --- |
| `控制中枢未运行` | service-manager 不可达。 | 点击“修复运行控制”。 |
| `正在修复` | 已开始检查和拉起控制中枢。 | 等待，不要重复点击。 |
| `核心服务异常` | service-manager 可用，但长期服务不健康。 | 点击“修复运行控制”。 |
| `修复完成` | 核心服务恢复。 | 刷新页面或重新进入入口。 |
| `需要人工处理` | 自动修复失败。 | 查看详情或交给 AI 排障。 |

提示文案应说明影响，例如：

```text
控制中枢未运行。pi-agent、SmallPhone 和 cc/codex 可能无法打开或显示旧状态。
点击“修复运行控制”，系统会检查控制中枢、重新注册核心服务并尝试恢复运行。
```

## 修复流程

运行修复建议按以下顺序执行：

1. 暂停前台自动保活的重复拉起，避免并发修复。
2. 轻量检测 service-manager health。
3. 如果 service-manager 不可达，尝试按安装后的启动入口拉起。
4. 读取 service-manager 配置，确认 token/config 存在但不打印内容。
5. 重新加载或同步服务注册。
6. 检查是否存在同名随机重复服务。
7. 确保稳定服务 ID 存在：`smallphone`、`pi-agent`、`cloudcli`。
8. 对不健康服务执行受控重启。
9. 检查端口和 HTTP health。
10. 刷新 Android UI 状态。
11. 输出脱敏修复摘要。

如果连续修复失败，必须停止自动重试并提示用户。不要在后台无限循环修复。

## 常见问题与修复方向

| 现象 | 优先判断 | 修复方向 |
| --- | --- | --- |
| 服务列表找不到 service-manager | 控制中枢未启动或配置缺失。 | 拉起 service-manager，检查配置路径。 |
| `pi-agent` 控制不了 | 服务未注册或 ID 不稳定。 | 重新注册稳定 ID，清理重复随机服务。 |
| `cloudcli` 打开旧端口 | 稳定 ID 指向旧服务定义。 | 更新 `cloudcli` 服务定义并重启。 |
| UI 显示运行但端口不可达 | pid 存在但服务不健康。 | 以 health 为准，重启服务。 |
| CPU 持续很高 | 修复/保活循环过快或服务崩溃重启。 | 降低重试频率，停止异常服务，展示详情。 |
| 日志里出现凭据 | 日志未脱敏。 | 停止输出敏感字段，生成脱敏摘要。 |

## 避免 CPU 拉满

运行修复必须节流：

- 同一修复入口一次只允许一个修复任务。
- service-manager 拉起失败后要退避重试。
- 单个服务连续失败后暂停自动重启，提示用户。
- 不在修复循环里执行全盘搜索、npm、apt、cargo 或长时间构建。
- 只有用户点“查看详情”时才读取较长日志。
- 日志 tail 限制行数。

建议退避：

| 失败次数 | 行为 |
| --- | --- |
| 1 | 立即轻量重试。 |
| 2 | 等待数秒后修复。 |
| 3 | 停止自动修复，提示用户。 |

## 日志与脱敏

运行修复可以记录：

- 修复开始和结束时间。
- 检查的服务 ID。
- 服务状态。
- 健康检查结果。
- 端口可达性。
- 最近错误摘要。

运行修复不得记录完整：

- API key
- token
- auth token
- provider secret
- service-manager auth token
- 用户输入的模型配置密钥

如果需要展示服务环境变量，必须过滤 `KEY`、`TOKEN`、`SECRET`、`AUTH`、`PASSWORD` 等字段，或只展示字段名不展示值。

## 给 AI 的处理原则

AI 排障时应按顺序：

1. 先读本文。
2. 再读 `SERVICE_MANAGER.md`。
3. 再读 `openhouse-runtime-policy.md`。
4. 如需分层恢复，再读 `TROUBLESHOOTING.md` 或 `RECOVERY.md`。

AI 不应直接重装环境。先恢复 service-manager，再恢复上层服务。需要执行命令时，优先使用已有 bootstrap、service-manager API 或注册脚本。

## 验收标准

- service-manager 不可达时，用户能看到“修复运行控制”入口。
- 修复过程不会要求普通用户进入终端。
- 修复能恢复控制中枢和默认长期服务。
- 修复能发现重复服务、端口错误和健康检查失败。
- 连续失败后停止自动重试并提示用户。
- 修复不会删除用户数据或模型配置。
- 修复日志和 UI 不泄露 `key/token`。
