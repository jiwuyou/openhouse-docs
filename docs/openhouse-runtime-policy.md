# OpenHouse 运行策略

本文定义 OpenHouse 安装完成后的默认运行策略。目标是降低普通用户上手成本：只要 App 在前台，核心能力就应该自动保持可用；需要暂时省资源时，用户使用“停止运行栈”；需要结束本次 OpenHouse 使用时，用户使用“全部退出 OpenHouse”。

## 核心目标

默认体验应该是：

```text
打开 App
系统自动确认控制中枢可用
系统自动保持核心服务运行
用户直接进入 pi-agent、SmallPhone 或 cc/codex
```

普通用户不应在首次使用阶段学习如何启动后台服务。服务控制可以存在于高级入口，但不应成为开始使用 Claude 或 Codex 的前置知识。

## 前台自动保活

App 处于前台时，Android 侧负责轻量检测 service-manager。推荐检测间隔为 15-30 秒。

检测策略：

- 单次检测只做轻量 health，不做全量扫描。
- 连续 1 次失败：记录状态，短延迟后重试。
- 连续 2 次失败：触发控制中枢修复或拉起流程。
- 连续 3 次失败：提示用户“控制中枢异常”，显示“修复运行控制”和“查看详情”。
- 修复动作必须节流，避免短时间反复拉起进程导致 CPU 拉满。
- 高级设置可以关闭“自动保持控制中枢运行”。关闭后，App 前台不再主动保活 service-manager，用户需要手动启动或修复运行控制。

service-manager 可用后，它负责检查默认长期服务：

- `smallphone`
- `pi-agent`
- `cloudcli`

如果某个长期服务未运行或健康检查失败，service-manager 应按服务配置启动或修复该服务。Android 侧只展示聚合结果和用户入口。

## 默认长期服务

默认长期运行集合：

| 服务 | 前台行为 | 失败后的处理 |
| --- | --- | --- |
| `service-manager` | Android 侧检测并拉起。 | 连续失败后进入运行修复。 |
| `smallphone` | service-manager 保持运行。 | 尝试重启，失败后提示兼容服务异常。 |
| `pi-agent` | service-manager 保持运行。 | 尝试重启，失败后提示首次配置助手异常。 |
| `cloudcli` | service-manager 保持运行。 | 尝试重启，失败后提示 `cc/codex` 入口异常。 |

如果 `pi-agent` 依赖 `pi-web` 进程，`pi-web` 可以作为内部服务被托管，但用户侧仍只看到 `pi-agent`。

## 新增服务规则

新增服务默认按长期服务处理，除非它明确标记为手动服务。

应该长期运行的服务：

- 用户固定使用的工作台。
- 本地 Web UI。
- MCP server。
- agent server。
- Android 控制桥接服务。
- 需要被侧边栏长期打开的服务。

应该手动运行的服务：

- 高 CPU 或高内存任务。
- 本地大模型推理。
- 一次性转换、构建、下载任务。
- 用户明确只想临时测试的服务。

每个新增服务必须在文档或注册信息中说明：

- 是否长期运行。
- 为什么需要长期运行。
- 健康检查方式。
- 停止方式。
- 日志位置。
- 失败时是否自动修复。

## CPU 与内存控制

前台自动保活不能变成高频轮询。

实现要求：

- Android 侧只检测 service-manager health，不频繁扫全进程。
- service-manager 自己按服务 health interval 检查服务。
- 同一服务启动失败后必须有退避策略。
- 失败重试要有最大频率限制。
- 日志 tail、端口扫描、进程扫描只在修复或详情页触发。
- 不在前台保活循环里执行重型命令、npm、apt、cargo、全盘搜索。

建议策略：

| 场景 | 行为 |
| --- | --- |
| App 刚进入前台 | 立即做一次轻量 health。 |
| App 持续前台 | 每 15-30 秒检测一次 service-manager。 |
| service-manager 正常 | 不做额外拉起，只读取聚合状态。 |
| 某服务失败 | 交给 service-manager 按节流策略修复。 |
| 多次修复失败 | 停止自动重试，提示用户手动修复。 |

## 前台、后台与全部退出

OpenHouse 运行状态分为四类：

| 状态 | 说明 |
| --- | --- |
| 前台可用 | App 在前台，自动保持核心服务运行。 |
| 后台策略 | 可以按产品设置决定继续运行或降低保活频率。 |
| 停止运行栈 | 用户暂时停止 OpenHouse 管理的长期服务和控制中枢，App 界面保留，本次会话暂停自动保活。 |
| 全部退出 OpenHouse | 用户明确要求停止 OpenHouse 运行栈、关闭 OpenHouse 界面，并请求关闭 Termux 前台服务和终端会话。 |

“停止运行栈”和“全部退出 OpenHouse”是资源兜底，不是普通返回键。用户点击停止运行栈后，系统应停止 OpenHouse 管理的长期服务和控制中枢，让 CPU 接近空闲，保留当前 App 界面，并在本次 App 会话内暂停自动保活。用户点击全部退出 OpenHouse 后，系统先停止运行栈，再关闭 OpenHouse 界面，并请求关闭 Termux 前台服务和终端会话。再次打开 App 时，再按前台策略恢复核心服务，除非用户已在高级设置中关闭自动保活。

## 用户可见状态

普通用户只需要看到简化状态：

- `运行中`
- `正在启动`
- `正在修复`
- `未运行`
- `需要处理`

高级详情可以显示：

- service-manager health。
- 默认长期服务状态。
- 端口 health。
- 最近错误摘要。
- 日志入口。

错误摘要不得包含完整 `key/token`、auth token 或 provider secret。

## 验收标准

- App 前台 15-30 秒内能发现 service-manager 是否可用。
- service-manager 异常连续失败后会修复或提示。
- App 前台时 `smallphone`、`pi-agent`、`cloudcli` 默认保持运行。
- 高级设置关闭自动保活后，App 前台不再自动拉起 service-manager。
- 新增长期服务能进入同一托管机制。
- 保活逻辑不会高频执行重型命令，避免 CPU 拉满。
- 用户不需要学习服务控制，也能开始使用核心入口。
- 用户点击“停止运行栈”后，前台保活在本次会话内暂停，核心服务被关闭。
- 用户点击“全部退出 OpenHouse”后，先停止运行栈，再关闭 OpenHouse 界面，并请求关闭 Termux 前台服务和终端会话。
- UI 和日志不泄露 `key/token`。
