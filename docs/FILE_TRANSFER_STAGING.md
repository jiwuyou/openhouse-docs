# 文件中转站和文件协作

OpenHouse 的文件能力目标是让用户和 AI 可以围绕同一批文件协作。用户可以上传、下载、复制路径；AI 可以读取、生成、整理文件，并把结果放到用户容易找到的位置。

本文描述产品原则和推荐路径。具体 UI 是否已经实现，以当前 App 和 service-manager 注册状态为准。

## 推荐目录

优先使用一个明确的中转目录：

```text
/root/openhouse/files
```

兼容或备用目录：

```text
/root/Downloads
/root/workspace
/sdcard/Download/OpenHouse
```

如果已经获得外部存储权限，并且用户希望用系统文件管理器或浏览器下载体验，应优先把用户可见文件放到外部目录，再在 Ubuntu 内通过软链接或明确路径访问。

示例：

```text
/root/openhouse/files -> /sdcard/Download/OpenHouse
```

是否创建软链接取决于权限、Android 版本和当前文件系统状态。AI 不应在未确认的情况下移动大量用户文件。

## 用户给 AI 文件

理想体验：

1. 用户点击文件入口。
2. 选择文件上传或移动到中转站。
3. 界面返回一句可复制的话，包含文件路径和任务提示。

示例：

```text
请读取这个文件并按我的目标处理：
/root/openhouse/files/example.pdf
```

AI 收到路径后，应先确认文件存在、大小、类型和是否适合处理，再继续。

## AI 给用户文件

AI 生成文件时，应：

1. 放到中转目录或用户指定目录。
2. 返回清晰路径。
3. 说明文件用途。
4. 如果是可下载内容，优先放在浏览器或系统下载容易访问的位置。

示例：

```text
我已经生成文件：
/root/openhouse/files/report.md

你可以在文件入口中打开，或复制这个路径交给其它工具继续处理。
```

## 文件处理原则

- 大文件先确认大小和类型，不要直接整份贴到聊天。
- 私密文件不要上传到外部服务，除非用户明确同意。
- 修改用户原文件前先备份，或写入新文件。
- 不要把 API key、token、cookie、私密日志写进公共文档或截图。
- 如果文件在 `/sdcard`，注意 Android 权限和路径兼容性。
- 如果需要 OCR、PDF 解析、图片处理或压缩，先检查本地工具是否存在，再决定安装。

## 给 AI 的简短规则

看到用户说“我上传了文件”“帮我处理这个文件”“把结果给我下载”时，优先：

1. 查找或询问文件路径。
2. 把工作目录限定在 `/root/openhouse/files` 或用户指定目录。
3. 处理前说明会生成什么。
4. 处理后返回路径和一句可继续交给 AI 的提示词。
